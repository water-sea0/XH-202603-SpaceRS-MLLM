# 自然语言输出的评估问题——同分异构与多答案场景

## 问题的本质

你描述的现象可以拆成三种情况，恰好对应了 NLP 评估的三个经典难题：

```
情况一：同分异构（语义相同，表达不同）
标准答案："图像左侧新建了一栋白色建筑"
模型输出："一栋白色楼房出现在了画面左边"
→ 意思一样，但字面重合度很低

情况二：一字之差，意思反转（语义相反，表达相似）
标准答案："变化区域位于图像中央"
模型输出："变化区域不位于图像中央"
→ 字面几乎一样，但差了一个"不"字，结论完全相反

情况三：多角度都正确（同一个输入，多个合理输出）
标准答案："图中有一架飞机停在停机坪上"
模型说A："机场停机坪上停着一架飞机"        ← 角度1：空间关系
模型说B："图片展示了一架大型白色客机"      ← 角度2：物体属性
模型说C："这是一张机场航拍图，画面主体是飞机" ← 角度3：整体场景
→ 三个都对，但都和标准答案不一样
```

这三个问题合在一起，意味着：**字面匹配度（如 BLEU/ROUGE）作为遥感描述类任务的评估指标，基本上是废的。**

---

## 为什么传统指标不行

### BLEU / ROUGE / METEOR 的问题

这些指标的核心思路是数 n-gram 重叠——标准答案和模型输出有多少个连续词相同。

```
标准答案："the new building is located on the left side of the image"
模型输出："on the left side of the image there is a new building"

BLEU-4 score ≈ 0.3~0.5  （只有短语片段匹配）
ROUGE-L ≈ 0.6           （最长公共子序列还行）
但两句话明明完全同义。
```

更糟糕的反例：

```
标准答案："a new building was constructed in the previously empty area"
模型输出A："a new building was not constructed in the previously empty area"  ← 多了一个 not，意思反转
模型输出B："in the previously empty area, a new building was constructed"    ← 只是语序调换

BLEU 打分：A ≈ B（因为 n-gram 重叠几乎一样）
实际质量：A 是错的，B 是对的
```

> **BLEU/ROUGE 的核心缺陷**：它们衡量的是**字符串相似度**，不是**语义相似度**。对于遥感描述这种"允许换说法、允许不同角度"的任务，毫无意义。

---

## 业界怎么解决

目前（2025-2026）主流方案有三类，按可靠程度从低到高排列。

---

### 方案一：LLM-as-Judge（用大模型当评委）

**做法**：拿 GPT-4o / Claude 等更强的模型，给它标准答案和模型输出，让它打分。

```
Prompt 示例：
"你是一个遥感图像描述评估专家。请比较以下两个描述，判断它们是否表达了相同的含义。
标准答案：{ground_truth}
模型输出：{model_output}
请给出 0-1 的相似度分数，并解释你的判断依据。"
```

**优点**：
- 能理解语义，不会被"语序不同但意思相同"骗到
- 能识别"差一个字但意思完全相反"
- 能判断"两个说法都对"的情况

**缺点**：
- 花钱（每次评估都要调 API）
- 不稳定（同一个 prompt 问两次可能得到不同的分）
- 自己也可能犯错（GPT-4 对遥感专业术语不一定理解准确）
- 评估 30 万条样本的成本不可忽略

**适用场景**：小规模人工抽检（比如从 30 万条里随机抽 500 条做质量评估）。

---

### 方案二：基于 Embedding 的语义相似度

**做法**：用一个文本 Embedding 模型（如 text-embedding-3-large、BGE-M3）分别给标准答案和模型输出编码成向量，算余弦相似度。

```
similarity = cosine_similarity(
    embed(ground_truth),
    embed(model_output)
)
```

**优点**：
- 比 BLEU 靠谱得多——Embedding 模型能捕捉语义而非字面
- 计算快，可以覆盖全量样本
- 不需要额外调 API

**缺点**：
- Embedding 模型本身可能偏向某种表达风格
- 对"差一个 not 就反转"这种细粒度语义差异，Embedding 的区分能力有限
- 阈值怎么定是个问题（>0.85 算对？>0.9？）

**适用场景**：全量训练数据的粗筛（过滤明显胡言乱语的样本）。

---

### 方案三：结构化分解评估（目前遥感领域最靠谱的做法）

**做法**：不评估整段话，而是把描述**拆成若干可验证的原子命题**，逐个判断对错。

举例——对于变化描述任务：

```
标准答案："在图片中央偏左的位置，新建了一栋白色建筑，原来的空地被建筑取代。"

分解为原子命题：
① 变化位置：图片中央偏左          → 坐标框可以验证
② 变化类型：新建（出现新物体）    → 可以通过"从无到有"验证
③ 变化对象：建筑                  → 需要判断物体类别是否匹配
④ 对象属性：白色                  → 需要判断颜色是否匹配
⑤ 变化前状态：空地                → 需要判断时相A该位置是否为空地
```

然后对每个原子命题打分，最后加权求和。

**遥感领域的天然优势**：遥感描述通常有**结构化标注**（如 VRSBench 的逐图 JSON 里有 `objects[{obj_cls, obj_coord}]` 和 `qa_pairs[{question, answer}]`），这些结构化数据可以作为评估的"锚点"。

```
实际做法：
1. 用 NER/关系抽取从模型输出里提取 (物体, 属性, 位置, 关系) 四元组
2. 和标准答案里的结构化标注做匹配
3. 匹配上的比例就是分数

→ 这样就不怕"同分异构"了——不管怎么换说法，提取出的四元组应该一样
→ 也不怕"一字之差"了——如果模型说的是"没有建筑"，提取不出建筑实体，得分为0
→ 也不怕"多角度都正确"了——只要提取的四元组和标准答案有交集就算分
```

**缺点**：需要额外的信息提取步骤；信息提取本身也可能出错；对非结构化的纯文本描述效果差。

**适用场景**：有结构化标注的数据集（VRSBench、XLRS-VG 的 bbox 标注、LEVIR 的 changeflag 标注），评估时可以用标注做锚点。

## 对我们的实际影响

### 训练阶段：问题不大

训练时，模型不需要"评估自己的输出"，它只是在做**下一个 token 预测**（next-token prediction）：

```
输入：  "<image>\n[optical 256 rgb][changecaption] 描述两张图中的变化："
标签：  "在图像左侧，新建了一栋白色建筑"
Loss：  -log P("在"|input) - log P("图"|input+"在") - log P("像"|input+"在图") ...
```

Loss 只看**和这条标准答案差多少**，不关心"有没有其他合理答案"。所以训练阶段：
- 标准答案不是唯一的 → 没关系，模型学到了"这个场景可以有这种表述"
- 同一张图有多个标注（如 LEVIR 每张图有 5 条 sentence）→ 更好，相当于数据增强，模型能学到更多表达方式

**训练只要数据量大、覆盖面广、标注质量还行就够了。**

### 评估阶段：需要认真对待

训练完要判断"模型好不好"，就需要解决你提出的问题。建议分两层：

| 层级 | 方法 | 覆盖范围 | 目的 |
|------|------|---------|------|
| **L1 粗筛** | Embedding 语义相似度 | 全量 | 过滤明显错误（胡言乱语、完全不相关） |
| **L2 精评** | LLM-as-Judge + 人工抽检 | 随机 500~1000 条 | 给出可信的质量分数 |
| **L3 结构化** | 原子命题分解（仅限有结构化标注的数据集） | VRSBench / XLRS-VG 等 | 精确评估定位/属性/类别准确性 |

---

## 一个具体例子：VRSBench 是怎么做的

VRSBench 的训练集同时提供了两种标注：

1. **自然语言标注**（LLaVA conversation format，用于训练）：
   ```
   GPT: "The image shows a white aircraft parked on the tarmac..."
   ```

2. **结构化标注**（逐图 JSON，可用于评估）：
   ```json
   {
     "caption": "The image shows a white aircraft...",
     "objects": [
       {"obj_cls": "airplane", "obj_coord": [0.61, 0.40, 0.35, 0.87], "obj_size": "large"}
     ],
     "qa_pairs": [
       {"question": "What color is the plane?", "answer": "White"}
     ]
   }
   ```

所以评估时可以：
- 对 **Caption** 任务 → 用 LLM-as-Judge（因为没有对错之分，只有好坏之分）
- 对 **Refer** 任务 → 直接算 IoU（预测的框和标注框的交并比），这是硬指标
- 对 **VQA** 任务 → 精确匹配或用 Embedding 相似度（因为答案通常很短，如 "White" / "3"）

> **核心思路**：**能硬评估的就硬评估，不能硬评估的才上 LLM 判分**。不要让模型去判断"描述写得好不好"，直接把描述里的关键信息抽取出来和结构化标注比对。

---

## 总结

| 你提的问题 | 根本原因 | 解决方向 |
|-----------|---------|---------|
| 同分异构（说法不同意思相同） | n-gram 匹配只看字面 | Embedding 语义相似度 / LLM-as-Judge |
| 一字之差意思反转 | n-gram 匹配对否定词不敏感 | 结构化分解（提取原子命题逐个验证） |
| 多角度都正确 | 自然语言的固有歧义 | 训练时用多条标注做数据增强；评估时与多条参考做软匹配 |

最后再强调一遍：**这些是评估阶段的问题，不影响数据预处理和训练。** 我们现在做数据集统一，给每条样本一个标准答案就行。模型训完之后怎么评，是另一个独立工程，到时候再搭评估脚本。

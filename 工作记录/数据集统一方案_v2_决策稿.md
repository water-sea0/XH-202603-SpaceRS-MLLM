# 数据集统一方案 v2（决策稿）

> 状态：第一轮本地数据盘点与模型资料核验后的建议稿。  
> 原则：原始数据只读；先统一语义和索引，再由模型适配层按需处理像素。

配套附件：

- `工作记录/rs-vl-1.0.schema.json`：机器可校验的 canonical record schema。
- `工作记录/数据集到RS-VL-1.0映射表.md`：逐数据集字段、split、任务和异常处理规则。

## 1. 结论先行

不建议把全部图片预先切成 448×448，也不建议为了 28 的倍数永久缩放原图。

建议采用三层结构：

1. **原始层**：保留现有 `dataset/` 下的文件、Arrow 分片和目录结构，不移动、不改名、不重编码。
2. **规范层（canonical manifest）**：只生成 JSONL 标注/索引，记录原图位置、尺寸、时相角色、任务、答案、坐标和来源 split。
3. **模型适配层**：
   - Qwen2.5-VL：运行时保持宽高比并由 processor 调整到 28px 的倍数；通过 `min_pixels/max_pixels` 控制视觉 token 预算。
   - EarthDial：运行时采用其 InternVL 系动态切块策略，448×448 tile + 全局 thumbnail，并限制 tile 数；不把整幅 8K 图机械切成上百张图全部输入。

这能同时满足：不同尺寸、28px 倍数、LEVIR（本工作区唯一双时相来源）、两种模型、原始数据不动，以及上云后按机器条件选择缓存策略。

## 2. 已核实的关键事实

### 2.1 模型输入机制

- Qwen2.5-VL 原生支持动态分辨率和多图输入。官方示例用 `min_pixels/max_pixels` 控制每图视觉 token 数；显式指定宽高时会取接近的 28px 倍数。
- 28 不是“数据集图片必须永久改成 28 的倍数”，而是 Qwen 图像 processor 的运行时约束。
- EarthDial 基于 InternVL2。论文的 Adaptive High Resolution 会选择合适的宽高网格，将图像变为 448×448 tile，并增加全局 thumbnail；论文写明训练时为 1-12 tiles、推理最多 40 tiles，表 3 的阶段训练配置使用 1-6 adaptive patch sizes。
- 因而“4096 图固定切 81 块、7360 图切几百块并全部作为多图输入”并不是 EarthDial 的做法，会导致视觉 token 爆炸且丢失全局-局部对应关系。
- GeoLLaVA-8K 也把超高分辨率的核心难题归因于 token explosion，并专门做背景 token 剪枝和锚定 token 选择；这进一步说明“全量均匀小切片全部输入”不适合作为基础方案。

- Qwen2.5-VL 官方说明：<https://qwenlm.github.io/blog/qwen2.5-vl/>
- EarthDial 论文：`reference & PPT/EarthDial.pdf`
- EarthDial 官方代码：<https://github.com/hiyamdebary/EarthDial>

### 2.2 本地数据实际情况

- `LEVIR-MCI` 与 `Levir-CC-dataset` 的 JSON 解析后完全相等，均为 10,077 对、每对 5 条描述，split 为 train 6815 / val 1333 / test 1929。
- 抽查的同名 A/B 图在两个目录中逐像素完全相同；文件 hash 不同只是编码/元数据差异。因此二者不能当作两个独立数据源重复加入训练或评测。应把 LEVIR-MCI 视为同一图像族的增强版本（额外有 label/mask）。
- LEVIR 的 `changeflag` 基本严格平衡：no-change 5039 / change 5038。
- VRSBench train 实测为 142,390 条：caption 20,264 / refer 36,313 / VQA 85,813，引用 20,262 张唯一图片。
- VRSBench 的 `Annotations_val/` 含 caption、objects、QA，可从官方 val 图片构造三类评测；不必把 train 再随机切成“测试集”。
- XLRS-Bench-lite 实际是 3,080 条英文多选 VQA，不是 caption 或 grounding 数据源。
- `XLRS-Bench_caption_zh` 与 `XLRS-Bench_visual_grounding_zh` 是独立数据集；后者有原生 train/test。
- 本地 `XLRS-Bench_caption_zh/train` 的 state 声明 23 个 Arrow shard，但实际只有 22 个，缺少 `data-00012-of-00023.arrow`；全量转换前要补齐或在审计报告中标记数据集不完整。
- MME-RealWorld 只取 Remote Sensing 子集是正确范围：英文 3,738 条、中文 300 条，任务为通用多选 VQA（color/count/position）。该来源必须出现在总任务统计与测试集配额中。英文 RS 引用的 1,265 个图中当前缺 208 个，先尝试补下载，仍缺失者只进入本地审计报告。

## 3. 规范层：不要直接把 LLaVA JSON 当唯一真相

LLaVA conversation 可以作为导出格式，但不适合做唯一 canonical 格式，因为：

- Qwen 与 EarthDial 的图像消息、task token、坐标表达并不相同；
- 五条 LEVIR caption 在训练时可展开成五条，评测时应是一条样本对应五个 reference；
- bbox 若只埋在字符串里，切图、缩放和评测时难以可靠变换；
- Arrow 内嵌图片不应为了生成 manifest 就全部解包复制。

建议 canonical JSONL 每行一个“图像组 + 任务实例”：

```json
{
  "schema_version": "rs-vl-1.0",
  "id": "levir/test/000001/changecaption",
  "source": "levir_mci",
  "source_split": "test",
  "task": "change_captioning",
  "language": "en",
  "media": [
    {"uri": "dataset/LEVIR-MCI/.../A/test_000001.png", "role": "t1", "width": 256, "height": 256},
    {"uri": "dataset/LEVIR-MCI/.../B/test_000001.png", "role": "t2", "width": 256, "height": 256}
  ],
  "prompt": "Describe the changes between time 1 and time 2.",
  "target": {
    "texts": ["reference 1", "reference 2", "reference 3", "reference 4", "reference 5"],
    "class_label": null,
    "boxes_norm_xyxy": []
  },
  "meta": {
    "sensor": "optical",
    "bands": ["r", "g", "b"],
    "changeflag": 1,
    "image_group_id": "levir/test/test_000001",
    "license_id": "source-defined"
  }
}
```

规则：

- 使用 JSONL，便于流式读取、增量生成和分片；不要把数十万条塞进一个 JSON 数组。
- `uri` 一律相对于一个可配置的数据根目录；不要写 `F:\...` 绝对路径。
- Arrow 内嵌图片用可解析 URI（例如 `arrow://相对分片#row=42&field=image[0]`）；云端训练时再物化到 cache。
- `media` 明确记录 `t1/t2`，不把双时相拼成一张左右图。
- bbox 在 canonical 层统一为浮点 `xyxy`、范围 `[0,1]`；导出 Qwen/EarthDial 时再转换为各自需要的字符串或坐标尺度。
- `sensor/bands` 作为元数据保留，但当前全为 optical/RGB 时不强制写进自然语言 prompt。
- 不把图像“像素宽度”误叫 spatial resolution。若无 GSD，字段名用 `width/height`；EarthDial 的 `[hr rgb 0.5]` 中 `0.5` 表示约 0.5 m GSD，不是 512px。

## 4. 两个模型的导出适配器

这里的 adapter 不是新的神经网络，也不是重新实现模型预处理。两个模型源码已经分别提供了 Qwen processor 和 EarthDial `dynamic_preprocess`。我们仍需写一层很薄的“翻译器”，因为模型源码不认识自定义的 RS-VL canonical JSONL。它负责：

1. 从普通文件或 Arrow 行取出图片；
2. 把 canonical 字段翻译成模型原生的消息/训练样本格式；
3. 调用模型自带的 resize/切块函数；
4. 在 canonical `[0,1]` bbox 与模型输出坐标之间换算；
5. 检查图片数量、占位符数量和视觉预算。

因此：**底层图像处理用模型自带代码；数据格式翻译和校验需要我们自己写。**

### 4.1 Qwen2.5-VL adapter

- 把 `media` 顺序转换为 chat content 中的多个 image item，再放 text item。
- 双时相 prompt 明示 `Image 1 = time 1, Image 2 = time 2`。
- 不预先 pad；processor 保持比例并调整到 28 的倍数。
- 建议预设两个 profile，并在每次训练/评测结果中记录 profile：
  - `smoke`: 每图 `max_pixels = 1280 * 28 * 28`（官方示例的成本/效果折中）。
  - `highres_eval`: 起步用 `4096 * 28 * 28`，显存不足再下调；不能只改数据而不记录视觉 token 预算。
- 双时相样本要限制“两图视觉 token 总预算”，避免每图都按单图上限导致 batch 显存翻倍。
- Qwen2.5-VL 官方倾向输出基于处理后/显示图尺寸的实际坐标；adapter 负责从 canonical `[0,1]` 转换和反转换，canonical 本身不跟随某个模型。

### 4.2 EarthDial adapter

- 使用 EarthDial/InternVL 自带的 `dynamic_preprocess` 思路，不离线生成固定 448 tiles。
- `max_num` 是“一张样本最多允许生成多少个 448×448 局部块”的上限。数值越大，模型看见的局部细节越多，但显存和时间也越高。训练 baseline 暂用总预算 6，能力评测可试 12；这不是原图数量。
- `global thumbnail` 是额外生成的一张 448×448 整图缩略图。局部块负责看细节，thumbnail 负责告诉模型这些细节属于整幅图的什么大环境。只有产生多个局部块时才需要它。
- 双时相举例：A 图动态处理后得到 3 个块，B 图得到 3 个块。送入模型时六个块会拼在一起，但还要同时告诉模型 `[3, 3]`：前三块属于 A，后三块属于 B。InternVL 把这个列表叫 `num_patches_list`。EarthDial 的 dataloader 已经会生成它；我们的 adapter 只需把 A/B 作为两张图正确传进去，不需要用户手写 `[3,3]`。
- EarthDial token 采用其论文/代码定义的格式。例如高分辨率 RGB 约 0.5m GSD 是 `[hr rgb 0.5]`，change detection 另有 `[changedet]`。不要自行发明 `[optical 7360 rgb]` 并称为 EarthDial 原格式。
- 当前数据多数没有可信 GSD；此时应使用普通 RGB/任务 prompt 或显式 `gsd=null`，不应由图片宽度伪造 GSD token。

代码核验补充：EarthDial 当前 `dynamic_preprocess` 默认 `image_size=448, max_num=6`，先将整图缩放到最接近原宽高比的有限网格，再裁成 448 块；它不是在原始像素坐标上用 448 滑窗。其双/多图 loader 对每张图使用 `max_dynamic_patch // num_image`，也就是多张图共享总 patch 预算，同时记录每张图的 tile 数。双时相 adapter 应复用这一语义。

## 5. 是否切图、缩放、补边

| 场景 | canonical 层 | Qwen adapter | EarthDial adapter |
|---|---|---|---|
| LEVIR-MCI（本工作区唯一双时相来源，2×256×256） | 两张原图，role=t1/t2 | 两个 image item；各自运行时调整到合法尺寸 | 两图分别处理；小图不切 |
| VRSBench 512×512 | 原图 | processor 运行时约束 28 倍数 | 通常 1 tile；不离线切 |
| 2K-8K caption/VQA | 原图 | 按 `max_pixels` 等比例缩放 | 动态宽高网格 + thumbnail，限制 tile 数 |
| 2K-10K grounding | 原图 + 原图归一化 bbox | 先做整图 baseline | 同上；不使用答案 bbox 选择输入 tile |

边缘处理原则：

- 初版不做固定 stride 的离线全图切片，因此不存在“最后剩 100px 怎么办”的主问题。
- 若后续实验确实需要滑窗，使用“覆盖到边界”的窗口起点：`min(i*stride, length-tile_size)`，让最后一个窗口向内对齐；不要丢弃边缘，也不要用黑边改变内容。
- overlap 只用于局部检测/grounding 实验，建议 10%-20%；caption/VQA 不默认滑窗。
- 不能用 ground-truth bbox 挑 tile 后再评测 grounding，这会泄露答案。

## 6. 20GiB 包：评测集与训练冒烟集逻辑分开

你说得对：如果只用少量 test 样本确认训练程序能否前向、反向和保存 checkpoint，随后彻底删除该次权重，并从原始干净 checkpoint 开始正式评测，那么不会对最终模型造成持续的数据污染。仍建议拆成两个 manifest，主要是防止脚本误把这些记录混进正式训练；二者共用一个包和目录，不会重复复制媒体。

```text
rs_vl_bundle_20g/
├── media/                 # 只复制被抽中的原始图，保持无损或原编码
├── manifests/
│   ├── smoke_train.jsonl  # 只来自源 train
│   ├── smoke_val.jsonl    # 只来自源 train/val，且与 smoke_train 图像组互斥
│   └── eval.jsonl         # 官方 test/val/benchmark 数据，永不与 smoke 权重混用
├── exports/
│   ├── qwen/
│   └── earthdial/
├── stats.json
└── README.md
```

缺图/坏图列表保存在工作区本地 `reports/audit_issues.jsonl`，默认不上传到 smoke/eval 包，也不会被模型读取。它只用于说明“哪些原始条目被排除以及原因”，便于补下载后增量恢复。

### 6.2 分层抽样与体积预算

- 硬限制按 `20 GiB = 20 * 1024^3 bytes` 检查，媒体目标约 18.5 GiB，至少保留 1.5 GiB 给清单、导出文件、文件系统误差和重试余量。
- 抽样单位是 `image_group_id`，不是 conversation。先选图，再附上该图有限数量的任务标注，避免同一张图被复制多次。
- 分层维度：`source × task × language × category × resolution_bin`；change detection 再按 change/no-change 平衡。
- 每图标注上限建议：1 caption、最多 3 VQA、最多 2 grounding；LEVIR 评测保留 5 个 caption references 但只预测一次。
- 去重/防泄漏至少按 `source image id + split`，关键来源再加像素 hash 或感知 hash。LEVIR 两目录必须映射为同一 `image_group_id`。
- 不能直接把全部候选混在一起做“谁小先选谁”的贪心。正确顺序是：先规定每个分层的数量目标和最低保障，再在层内随机选图，最后用真实文件字节做上限修正。

本地真实文件字节（GiB，按 `1024^3`）表明旧估算需要调整：

| 媒体来源 | 文件数 | 总体积 | 平均单文件 |
|---|---:|---:|---:|
| LEVIR-MCI test A+B | 3,858 | 0.510 GiB | 约 0.136 MiB |
| VRSBench val | 9,350 | 3.879 GiB | 约 0.425 MiB |
| VRSBench train | 20,264 | 8.272 GiB | 约 0.418 MiB |
| MME remote_sensing | 1,087 | 49.027 GiB | 约 46.2 MiB |
| XLRS-lite Arrow | 74 shards | 35.566 GiB | 约 492 MiB/shard |
| XLRS caption zh Arrow（当前 22 shards） | 22 shards | 9.790 GiB | 约 456 MiB/shard |
| XLRS grounding zh train/test | 各 64 shards | 各约 59.1 GiB | 约 945 MiB/shard |

因此，不能把 MME RS 抽 300 图：仅按当前平均值就约 13.5 GiB，单独已经超限。也不要按 Arrow shard 抽样，一个 shard 接近 0.5-1 GiB且内部类别可能不均衡；应从 Arrow 中选择行并只物化被选中的图片。

建议首版配额如下。空间是按本地平均文件字节估算，Arrow 行在实际物化后以真实字节重新核算：

| 来源/划分 | 图像组目标 | 标注目标 | 估算空间 |
|---|---:|---|---:|
| LEVIR-MCI test | 全部 1,929 对 | 1,929 change caption（各5 references）+ 1,929 change detection；其中 no-change 965 / change 964 | 0.510 GiB |
| LEVIR-MCI train smoke | 200 对 | 两任务各200；change/no-change 各100 | 约 0.053 GiB |
| VRSBench val | 3,500 图 | 3,500 caption；目标约5,000 grounding、10,500 VQA（每图分别最多2/3条） | 约 1.45 GiB |
| VRSBench train smoke | 750 图 | 750 caption；目标约1,000 grounding、2,250 VQA | 约 0.31 GiB |
| XLRS-lite benchmark | 350 行/图 | 350 英文多选 VQA；13个 category 先各取最低数，再平均补齐 | 约 4.1 GiB |
| XLRS caption zh | 300 dev + 75 train smoke | 375 image caption；按 image_group 分开 | 约 1.2 GiB |
| XLRS grounding zh | 400 test + 80 train smoke | 480 grounding；按 category 和目标框面积分层 | 约 4.1 GiB |
| MME-RS EN | 100 图 | 目标约300 VQA：color/count/position 各100 | 约 4.57 GiB |
| MME-RS ZH | 40 图 | 目标约120 VQA：color/count/position 各40 | 约 2.17 GiB |
| **合计** |  | 五类任务、全部有效数据来源 | **约 18.47 GiB** |

MME 一张图可能对应三类问题，所以“100 图”可以产生约 300 条平衡 VQA；媒体只复制一次。若某图没有所需类别，就继续抽下一图补足标注目标。

具体抽样算法：

1. 建立候选表，只记录 `image_group_id、source、split、task、language、category、changeflag、answer_letter、bbox面积档、文件字节`。
2. 为每个分层设置 `minimum` 和 `target`。例如 LEVIR change/no-change 必须各半；MME 按 EN/ZH 后再按 color/count/position，并检查正确选项字母不过度偏斜；XLRS-lite 按13个 category，并检查答案字母；grounding 按 category × 小/中/大目标框。Caption 没有正负样本，只平衡来源、语言和尺寸档。
3. 使用固定随机种子在每层抽到 `minimum`，先保证任何一层都不会缺席。
4. 再以轮询方式给各层一次增加一个 image group，直到达到 `target`，而不是让大数据集先吃完预算。
5. 同一图片在多个任务中只计一次媒体字节；附带标注仍受每图上限控制。
6. 若超过 18.5 GiB，只从“已超过 minimum 的层”中移除最大文件；绝不削减到 minimum 以下。若还有空间，再按轮询补样本。
7. 输出最终的计划数、实际数、缺额、真实字节和随机种子，任何配额未达到都报错而不是悄悄继续。

### 6.3 评测汇总

评测不是把五类任务硬凑成一个神秘总分，而是先分别回答五个直白问题：

| 任务 | 怎么判断模型答得对不对 |
|---|---|
| Change Detection | 模型回答 change/no change，统计答对百分比。 |
| VQA | 多选题比较选项字母；普通短答案忽略大小写和多余空格后比较。统计答对百分比。 |
| Visual Grounding | 比较模型框和标准框的重叠程度。IoU=1 表示完全重合，0表示不相交；IoU≥0.5 可记为一次定位成功。 |
| Image Caption | 生成描述与人工参考描述比较；自动分数只能辅助判断，另固定抽一小批人工看是否事实正确。 |
| Change Captioning | 与 LEVIR 为该图提供的5条参考描述一起比较；模型每个图像对只回答一次。 |

“macro average”只表示：先算各来源/类别自己的平均成绩，再让它们等权平均。例如 VRSBench 有一万道 VQA、MME只有几百道，不能让前者凭题多把后者完全淹没。首版报告保留各任务独立分数；总分只作为方便排序的附加值。

每次结果还要记下模型版本和图像处理档位，否则“模型A用了更多高分辨率块、模型B用了低分辨率”时，分数无法公平解释。

### 6.4 Arrow 数据怎么上传

分两种情况：

1. **20GB 小包**：不能为了抽中一行就带上约 0.5-1GiB 的整个 Arrow shard。采样程序从入选行取出原始图片编码字节，按内容 hash 命名保存到 `media/xlrs_*/`；不缩放、不切图、不重复转 JPEG。manifest 改为普通相对文件路径。这样包是自包含的，服务器不需要原 Arrow 才能跑。
2. **以后上传全量数据**：原始 Arrow shard 保持原样上传，canonical manifest 保留 `shard + row + column + index` 定位。服务器首次使用时可流式读取 Arrow，或者一次性物化成可删除的 image cache。训练框架若不支持直接读 Arrow，就由 adapter/cache builder 转换；不需要在本地先复制几百GB图片。

Arrow 文件本身不符合 Qwen 或 EarthDial 的直接输入格式；它只是存储容器。真正交给模型的是 adapter 从对应行解出的图片和文字。

## 7. 实施顺序

### 阶段0：冻结规则，不处理图片

- 确认 RS-VL schema、MME只取RS、LEVIR去重、20GB配额和 D3 运行时档位。
- 输出：schema、映射表、配置文件。
- 通过条件：每个来源、任务、split、坐标格式都有唯一明确规则。

### 阶段1：只读审计原始数据

- 扫描 JSON/Arrow 元数据和图片头，不修改文件。
- 统计真实行数、尺寸、字节、任务、类别、缺图、坏图、重复图和 split 交叉。
- 尝试补齐 MME-RS 缺图和 XLRS caption 缺 shard；补不齐的写入本地 `reports/audit_issues.jsonl`。
- 输出：`inventory.json`、`audit_issues.jsonl`、统计 Markdown。
- 通过条件：所有“应该存在但不存在”的文件都有明确列表。

### 阶段2：生成 canonical manifest

- 分别编写 LEVIR、VRSBench、三套XLRS、MME-RS converter。
- 普通图片只记录相对路径；Arrow 图片记录 shard、row、column，不解包、不切图。
- 输出：每来源每split一个 JSONL，以及合并索引。
- 通过条件：转换只新增标注文件，原始数据目录 hash/数量不变。

### 阶段3：严格验证

- 校验 schema、全部媒体可解析、A/B成对、bbox顺序、答案非空、ID唯一。
- 检查 LEVIR两目录重复、训练/评测 image_group 不交叉、任务统计是否符合原始数据。
- 对 bbox 做“canonical→模型坐标→canonical”往返测试，误差应接近0。
- 通过条件：valid manifest 零缺图、零非法框、零 split 泄漏；问题只留在本地审计报告。

### 阶段4：实现两个薄 adapter

- Qwen adapter：读取 canonical、构造多图 chat content、调用官方 processor、处理28倍数和坐标换算。
- EarthDial adapter：读取 canonical、把 LEVIR A/B交给官方多图 loader、调用 `dynamic_preprocess`、传递每图块数。
- 每个 adapter 用单图、小图、8K图、双时相、grounding 各做至少一个测试。
- 通过条件：模型实际收到的图数与标注一致；不会把内部 tile 错当成独立语义图片。

### 阶段5：制作20GB包

- 按6.2的分层算法选 image group。
- 普通文件复制原编码；Arrow只提取入选行的原始编码图片，不复制整块1GB shard，也不切成448 JPEG。
- 生成 `smoke_train/smoke_val/eval` manifest；同一媒体文件只存一份。
- 通过条件：总大小≤20GiB；每层达到 minimum；实际任务和字节报告与文件一致。

### 阶段6：上云冒烟测试

- 先跑“数据加载→前向→反向→保存→重新加载”几个 step，不追求模型变好。
- 可以用 eval 中极少量样本做一次性管线探针，但该权重立即删除；正式 eval 从干净 checkpoint 开始。
- Qwen 和 EarthDial 都要分别跑通单图与 LEVIR双图。
- 通过条件：无OOM、无图片占位符错误、loss有限、checkpoint可重载。

### 阶段7：正式评测和全量上云

- 用干净模型按6.3分别评五类任务。
- 根据小目标 grounding 表现和显存占用决定 D3 的高分辨率档位。
- 全量数据上传时保留原始 Arrow/图片布局；云端按需建立可删除 cache。cache 失败可重建，canonical manifest 不变。

### 7.1 现有 `script/` 不应直接续跑

旧脚本可参考字段解析，但当前输出不应作为新方案基线，原因如下：

- `config.py` 写死本机绝对路径，无法上云迁移。
- `tiler.py` 对所有大图做原始像素 448 无重叠切片、黑色补边并转 JPEG；这既不等于 EarthDial 动态切图，也会引入不可逆重编码和大量文件。
- `token_prefix.py` 将像素宽度伪装为 EarthDial 的 spatial resolution/GSD token。
- LEVIR-MCI 与 Levir-CC 被重复转换和合并，造成图像与标注重复、评测泄漏。
- Arrow converter 使用 `read_all().to_pydict()`，会把含大图字节的整个分片载入内存；应逐 batch/逐行或 memory-map 流式读取。
- `convert_xlrs_caption_zh.py` 用 tile 数反推原图分辨率，不可靠。
- XLRS grounding 把整图 bbox 与没有空间索引的独立 tile 列表混在一起，模型无法知道每个 tile 对应原图哪里。
- MME 缺图时仍生成看似正常的训练条目，且仅用文件 stem 做 cache key，存在重名碰撞风险。
- `validate.py` 只检查第一张图片是否存在，没有检查全部媒体、`<image>` 数量、双时相配对、bbox、重复与 split 泄漏。

建议保留旧脚本不动作为历史记录，新建 `script/rs_vl/` 实现 v2；验证稳定后再决定是否归档旧版本。

## 8. 当前需要确认的决策

### D1（已接受）：20GiB 包逻辑拆分 smoke 与 eval

- 同一个包、同一个 media 目录，不产生重复媒体开销。
- eval 可选少量条目做一次性管线探针，但该次权重删除；正式评测从干净 checkpoint 开始。

### D2（已接受）：采用“原图 + canonical manifest + 两个 adapter”

- **建议：采用三层方案。** 它与两个模型的原生预处理都兼容，且不会把 480GB 数据再膨胀一份。
- 固定 tiles 只保留为后续可复现实验 profile，不作为唯一数据格式。

### D3（无需现在拍板）：模型运行时愿意花多少显存看高分辨率细节

- 这正是此前“该不该切、切多少”的问题，但现在把它放在 adapter 运行参数里，而不是永久写死在数据集里。
- 建议先提供两档：Qwen `1280` 与 `4096` 个视觉 token 上限；EarthDial 总 `max_num=6` 与 `12`，都启用 thumbnail。
- 上云后用一小批8K VQA/grounding比较：是否OOM、速度、定位准确率。选择显存能稳定承受且效果更好的档位。
- 无论选哪档，canonical 原图和标注都不用重做。因此 D3 可以等知道GPU型号/显存后再决定。

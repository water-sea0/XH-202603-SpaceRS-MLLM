1 多模态大模型（MLLM）训练数据长什么样？

在写转换脚本之前，先搞懂目标格式：一条"训练样本"到底包含什么。

📝 核心概念：一条「对话样本」（Conversation Sample）

多模态大模型的训练数据，本质上就是"图文对话记录"。每条样本 = 一张(或多张)图 + 一段对话。

目前社区最通用的格式是 LLaVA Conversation JSON，下面是它的结构解剖：

{
  "id": "vrsbench/train/00042",          ← 样本唯一ID（数据集/划分/序号）
  "image": "Images_train/00002_0000.png",   ← 图片的相对路径
  "conversations": [                           ← 对话内容，每条是一个说话回合
    {
      "from": "human",                        ← 谁在说话（human / gpt）
      "value": "<image>\n[caption] 描述这张图。"  ← <image> 是图片占位符
    },
    {
      "from": "gpt",
      "value": "图中显示一个收费站..."        ← 期望模型输出的回答
    }
  ]
}

训练时，模型看到 <image> 就知道这里有一张图，看到 [caption] 就知道要做什么任务。它学习的是：给定图片 + 指令 → 生成正确回答。

🎯 不同任务的数据格式差异

任务输入输出示例
Caption 单张图一段描述文字 "图中有一座桥梁横跨河流..."
Referring 单张图 + 物体描述坐标框 (bbox) "{<25><40><33><60>}"
VQA 单张图 + 问题简短答案 "3 辆车" / "是" / "城市"
ChangeCaption 两张图（前后时相）一段变化描述 "新建了一栋建筑，原来空地上..."
ChangeDet 两张图变化/无变化 (+ mask) "change" / "no change"
💡 关键理解：无论什么任务（描述/问答/定位），最终都统一为"图+文→文"的形式。 变化检测的特殊之处在于输入是两张图（时相A + 时相B），输出在某些情况下可以是 mask 图，但 MLLM 通常用文字表述变化内容。
2 EarthDial 是怎么统一异构数据的？

EarthDial 是发表在顶会上的一篇遥感多模态大模型论文，它的核心贡献就是一套异构数据统一框架。我们参照它的设计来做数据预处理。

🔑 创新一：统一 Token 体系

每条样本前面都带上两个特殊 token：模态 token（标明传感器/分辨率/波段）+ 任务 token（标明要做什么任务）。这样，一个 LLM 就能看懂"这张图从哪来、我要干什么"。

Token 前缀的结构

[optical 512 rgb] [caption]
 ↑                  ↑
 模态token           任务token
 (传感器 分辨率 波段)  (做什么)

实际样本举例

# 变化描述 (LEVIR)
[optical 256 rgb][changecaption]
Describe the changes...

# 视觉定位 (XLRS, 大图)
[optical 7360 rgb][refer]
请定位图片右下角的蓝色汽车...

# VQA (中等分辨率)
[optical 512 rgb][vqa]
图中收费站的颜色是什么？

🧩 创新二：Adaptive High Resolution（自适应高分辨率切图）

遥感图像的尺寸差异极大——小的 256×256，大的 11500×7500。如果粗暴缩放到同一尺寸，小图会失真，大图会丢失细节。
EarthDial 的做法：把大图切成 448×448 的小块（tile），训练时取 1~12 块，推理时最多 40 块。

我们的处理：LEVIR (256×256) 和 VRSBench (512×512) 不切；XLRS 系列 (4096+) 和 MME-RealWorld 遥感子集 (最大 11500) 需要切成 448×448 tile。

🔬 创新三：Data Fusion（多模态数据融合）

对于多光谱/SAR/多时相这类"非标准RGB"输入，EarthDial 的做法是：每 3 个通道分一组过 ViT 提特征，然后用双线性插值压缩聚合，统一映射到与 RGB 相同的视觉 token 空间。

⏸️ 本次暂不实现：我们目前的数据集都是 RGB/灰度图，没有 SAR 或多光谱数据。Data Fusion 模块留到后续有相关数据时再做。当前脚本只处理 Token 前缀生成 和 切图 两个环节。

📈 附加知识：三阶段渐进训练（论文中的训练策略）

阶段训练什么冻结什么用什么数据目的
Stage 1 ViT + MLP + LLM 全部 无 只有单图 RGB 建立基础视觉-语言对齐能力（最简单场景）
Stage 2 LLM + MLP ViT RGB + 时序数据 让模型理解"时间变化"（但不动视觉编码器）
Stage 3 LLM + MLP + DataFusion ViT + 多光谱 / SAR 引入更复杂的传感器输入

核心逻辑：先用最简单的 RGB 建立基础能力 → 再逐步扩展到时序和多光谱 → 避免一次性训所有数据导致模型难以收敛。
这是训练阶段的策略，我们当前只做数据预处理，不需要管这一层。

📦 四种原始存储格式，一次看清楚

格式①：LEVIR 标准 JSON（变化描述专用）

{
  "images": [
    {
      "filepath": "train",
      "filename": "train_000042.png",
      "imgid": 41,
      "changeflag": 1,                ← 0=没变化, 1=有变化
      "split": "train",
      "sentences": [                    ← 每张图有5条变化描述
        {"raw": "a new building appears...", "sentid": 205},
        ...
      ]
    },
    ...
  ]
}

格式②：LLaVA Conversation JSON（VRSBench）

[
  {
    "id": "Final_Data/v1.2",
    "image": "00002_0000.png",
    "conversations": [
      {"from": "human",   "value": "<image>\n[caption] Describe..."},
      {"from": "gpt",     "value": "The image shows..."}
    ]
  },
  { ← 同一张图，可以有多条不同任务
    "id": "Final_Data/v1.2",
    "image": "00002_0000.png",
    "conversations": [
      {"from": "human",   "value": "<image>\n[refer] location for <p>toll station</p>?"},
      {"from": "gpt",     "value": "{<45><45><59><59>}"}  ← 归一化坐标
    ]
  }
]

格式③：HuggingFace Arrow（XLRS 系列）

这是一种二进制列式存储格式，图片的 JPEG 字节直接嵌在 Arrow 文件里。需要用 Python 的 pyarrow 或 datasets 库读取。

数据集分片数每片大小Schema（列名）
XLRS-lite74~500MB path, index, question, options[], answer, category, l2-category, image[{bytes, path}]
XLRS-caption-zh22~100-900MB question_id, question, answer[], id, image{bytes, path}
XLRS-VG-zh64+64~800MB-1GB image_width, image_height, question_id, image{bytes, path}, question, answer, bbox[], category

格式④：MME-RealWorld JSON（多选题 VQA）

{
  "Question_id": "Perception/.../0000",
  "Image": "remote_sensing/xxx.jpg",
  "Text": "What objects are visible?",
  "Answer choices": ["(A) car & tree", "(B) ...", ...],
  "Ground truth": "D",               ← 正确答案是选项字母
  "Task": "Perception",
  "Subtask": "remote_sensing"
}
4 我们要做什么：从「各自为政」到「统一格式」

核心思路：每个数据集写一个转换脚本，把它的"原样"翻译成统一的 LLaVA Conversation JSON + EarthDial Token 前缀。

🔄 转换全景图：Before → After

❌ Before：LEVIR-MCI（JSON）

{
  "filepath": "train",
  "filename": "train_000042.png",
  "changeflag": 1,
  "sentences": [{
    "raw": "a new building appears"
  }]
}

问题：单张合并图，没有token前缀，不兼容其他数据集

✅ After：统一格式

{
  "id": "levir_mci/train/0042",
  "images": ["levir_mci/train_000042.png"],
  "conversations": [
    {"from":"human",
     "value":"<image>\n[optical 256 rgb][changecaption]
              Describe the changes..."},
    {"from":"gpt",
     "value":"a new building appears..."}
  ],
  "task": "changecaption",
  "sensor": "optical",
  "resolution": 256,
  "bands": "rgb",
  "source": "levir_mci",
  "language": "en",
  "split": "train"
}

优势：统一schema + token前缀 + 元数据 + 兼容LLaMA-Factory

❌ Before：XLRS Visual Grounding（Arrow）

Arrow Table (64 shards)
Schema:
  image_width: 7360
  image_height: 4912
  question: "给定一个7360x4912像素..."
  bbox: [xmin, ymin, xmax, ymax]
  image: {bytes: <JPEG binary>, path: "..."}

问题：二进制格式，图片嵌在文件里，坐标系统不统一

✅ After：统一格式 + 切图

{
  "id": "xlrs_vg/train/02950",
  "images": ["xlrs_vg/DOTA_v2_P5729_0_0.jpg",
             "xlrs_vg/DOTA_v2_P5729_0_1.jpg",
             ... ← 81个tile ],
  "conversations": [
    {"from":"human",
     "value":"<image>\n[optical 7360 rgb][refer]
              请定位右下角蓝色汽车..."},
    {"from":"gpt",
     "value":"bbox: [0.82,0.73,0.87,0.78]"}
  ],
  "task": "refer",
  "resolution": 7360,
  "language": "zh",
  "source": "xlrs_vg"
}

优势：图片已提取+切分，坐标归一化，中文token标注

❌ Before：MME-RealWorld（JSON 多选）

{
  "Question_id": "remote_sensing/0000",
  "Image": "remote_sensing/image.jpg",
  "Text": "What is visible?",
  "Answer choices": ["(A) ...", "(B) ...",
                     "(C) ...", "(D) ..."],
  "Ground truth": "D",
  "Subtask": "remote_sensing"
}

问题：多选格式特殊，图片超大（11500px），遥感子集与其他混在一起

✅ After：统一格式（仅遥感子集）

{
  "id": "mme_rw/train/0000",
  "images": ["mme_rw/image_r0_c0.jpg", ...],
  "conversations": [
    {"from":"human",
     "value":"<image>\n[optical 11500 rgb][vqa]
              What is visible?\n(A)...\n(B)...\n(C)...\n(D)..."},
    {"from":"gpt",
     "value":"D"}
  ],
  "task": "vqa",
  "resolution": 11500,
  "source": "mme_realworld"
}

优势：多选选项直接写入human消息，答案只保留选项字母

🎯 每个转换脚本干什么活（一句话概括）

脚本输入主要动作复杂度
convert_levir_mci.py JSON + 图片文件 读JSON → 每条sentence生成一条conversation → 生成token前缀 → 输出 ⭐ 简单
convert_levir_cc.py JSON + A/B分目录图片 读JSON → 匹配A/B文件对 → 双图放入images列表 → 生成token前缀 ⭐ 简单
convert_vrsbench.py LLaVA JSON + 图片 解析已有LLaVA格式 → 替换任务标签为EarthDial token → 坐标归一化 ⭐⭐ 中等
convert_xlrs_lite.py Arrow 分片（二进制） 流式读Arrow → 提取内嵌JPEG字节存盘 → 切4096大图为tile → 生成token前缀 ⭐⭐⭐ 较难
convert_xlrs_caption_zh.py Arrow 分片（二进制） 流式读Arrow → 提取图片 → 按实际分辨率切图 → 中文标注 ⭐⭐⭐ 较难
convert_xlrs_vg.py Arrow 分片（120GB） 流式读Arrow → 提取超大图 → 切大量tile → bbox坐标归一化 ⭐⭐⭐ 较难
convert_mme_realworld.py JSON + 超大图片 过滤遥感子集 → 将多选选项格式化写入human → 切超大图 ⭐⭐ 中等
merge_datasets.py 上面所有产出 按split拼接 → 任务平衡采样 → 输出train/val/test.json ⭐ 简单
5 脚本架构与执行流程

10个Python脚本，按依赖顺序执行。底层工具函数复用，避免重复代码。

📁 文件结构

scripts/
├── config.py              ← 全局配置（路径、token映射、切图参数）
├── utils/
│   ├── token_prefix.py   ← 根据传感器/分辨率/波段/任务生成token前缀

├── convert_levir_cc.py     ← ② 变化描述（A/B分目录）
├── convert_vrsbench.py     ← ③ Caption+Refer+VQA（LLaVA格式来源）
├── convert_xlrs_lite.py    ← ④ 复杂推理VQA（Arrow→JSON）
├── convert_xlrs_caption_zh.py ← ⑤ 中文Caption（Arrow→JSON）
├── convert_xlrs_vg.py      ← ⑥ 视觉定位（Arrow→JSON，含坐标）

🔗 执行顺序（依赖关系）

第1步：基础设施

config.py → token_prefix.py → image_utils.py：所有脚本都依赖这些底层模块

第2步：切图引擎

tiler.py：XLRS 系列和 MME 大图都需要切 448×448 tile

第3步：简单数据集（验证流程）

convert_levir_mci.py + convert_levir_cc.py：标准JSON+小图，最简单，先跑通

第4步：核心数据集

convert_vrsbench.py：数据量最大（14.2万条），覆盖3种任务，确立转换范本

第5步：Arrow数据集

convert_xlrs_lite.py → convert_xlrs_caption_zh.py：Arrow流式读取+切图

第6步：最复杂数据集

convert_xlrs_vg.py：120GB数据+超大图切分+bbox坐标归一化

第7步：多选VQA

convert_mme_realworld.py：多选格式处理

第8步：合并与校验

merge_datasets.py → validate.py：拼成最终产出+质量检查
